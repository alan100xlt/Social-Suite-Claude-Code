Universal Remediation System

How we make the platform self-healing — automatically detecting, diagnosing, and fixing errors across every edge function, webhook, and API call.

The Problem Today

We have 41 edge functions, but error handling is inconsistent. Some log errors, some don't. Some retry, most don't. When something breaks at 3am, nobody knows until a user complains.

Gap 1

The watchdog is blind to inbox failures

inbox-sync logs errors as "partial success" — but the watchdog only looks for "error". It never triggers remediation for the most failure-prone function.

Gap 2

Crashes vanish silently

If inbox-sync crashes entirely, it returns HTTP 500 but writes nothing to the health log. The watchdog still sees the last successful run. Nobody knows.

Gap 3

No Slack alerts for inbox

The alerting system exists (CronMonitor) but inbox-sync doesn't use it. Zero notifications for inbox pipeline failures, ever.

Gap 4

Retry = re-process everything

When the watchdog does retry a job, it re-syncs ALL companies. One broken company causes unnecessary load on 20 healthy ones.

The Solution: Four Layers

Instead of fixing just inbox, we build a universal system that any edge function can plug into.

Layer 1 — Shared Utilities

The building blocks every function uses

Three shared modules that replace copy-pasted code across the codebase:

ModuleWhat it doesReplaces
fetchWithRetry() Wraps any external HTTP call with automatic retry. Handles 429 rate limits (waits), 5xx server errors (retries), timeouts (retries). Backs off exponentially. Ad-hoc retry loops in 2 functions, zero retry in 39 others
logApiCall() One-line logging to api_call_logs with function name, duration, success/fail, company context. Copy-pasted logging code in 9 functions
diagnoseError() Takes any error string, returns: what went wrong, why, how to fix it, and whether it can fix itself. The client-side diagnosis engine in WebhookHealthTab (now shared)
Layer 2 — Remediation Engine

The brain that decides what to do about errors

A new edge function (edge-remediation) + database table that tracks every error across the platform.

When an error happens: The failing function sends a fire-and-forget message to the remediation engine. The engine diagnoses it and decides:

Error TypeExampleAutomatic Action
Auto-resolve Duplicate database entry Mark resolved immediately. These are benign.
Auto-retry Rate limit (429), timeout, server error (503) Queue for retry. Re-sync just the failed company in 10 minutes. Retry up to 3 times.
Escalate Auth failure (expired API key), database schema error Send Slack alert immediately with the reason and exact fix steps. Needs a human.
Exhausted retries Timeout that keeps happening after 3 retries Stop retrying, escalate to Slack. Something deeper is wrong.

The retry is targeted, not brute-force. Instead of re-running the entire inbox-sync (which processes all 20 companies), the remediation engine calls inbox-sync with just the one failing company ID. Healthy companies are untouched.

Layer 3 — Wiring

Connecting existing functions to the system

Each function gets a one-line fire-and-forget call after errors. This is not a rewrite — it's adding a single POST request at the end of error handling. The function's behavior doesn't change at all.

Priority order: inbox-sync (most failure-prone) > analytics-sync > rss-poll > getlate-* proxies

Layer 4 — Visibility

The Operations Center shows everything in real-time

The Webhook Health tab becomes a "Remediation" tab showing the queue: what's open, what's being retried, what's been escalated, what resolved itself. One-click manual resolve for anything.

Stress-Testing the Design

Here are the hard questions and honest answers about where this could break.

Is one edge function managing all remediation a single point of failure?

No, because it's decoupled. If edge-remediation itself goes down, the inbox/analytics/rss functions continue working normally — they just fire-and-forget the diagnosis call and move on. Errors would pile up in the log table but nothing breaks. The existing watchdog also monitors the remediation cron job itself, so it would retrigger it. The remediation function is also stateless — it reads from the database each run, so there's no state to lose.

Will it work for things beyond edge functions — like webhooks, client-side errors, third-party API failures?

Yes, with one addition. The diagnose action accepts any error string + function name + optional company ID. A webhook handler, a React error boundary, or even a PostHog webhook destination can POST to it. The diagnosis engine pattern-matches the error text regardless of source. The table has a generic function_name field, not an "edge function" field — it can track "posthog-client-error" or "slack-webhook" just as well.

What if the remediation retry makes things worse? (cascading retries, retry storms)

Three safeguards: (1) Max 3 retries per error, then it stops and escalates. (2) Deduplication — the same error for the same company only gets one active entry, not N entries per cron run. (3) The 10-minute interval creates natural backoff. A rate-limited company gets retried at 10, 20, 30 minutes — not in a tight loop.

Will the remediation table grow forever?

No. Resolved/escalated entries are historical data we can clean up on the same 30-day schedule as cron_health_logs. Only open and retrying entries are active. The dedup index ensures there's at most one active entry per unique error pattern per company.

How does the retry know how to re-invoke the original function?

A simple registry. Each function has a registered retry strategy: inbox-sync gets called with { companyId }, analytics-sync gets called with { companyId }, rss-poll gets called with {} (full re-run, no single-target mode). User-initiated functions like getlate-inbox don't get auto-retried — the user will retry themselves. New functions opt in by adding one entry to the registry.

What Else Can We Extend This To?

PostHog Client-Side Error Capture

PostHog already captures JavaScript exceptions automatically (capture_exceptions: true is enabled). We can create a PostHog webhook destination that forwards $exception events to our edge-remediation function. Every unhandled React error, failed API call, or runtime crash on the client would flow into the same remediation queue — diagnosed, triaged, and visible in the Operations Center alongside server errors. We'd also add a React Error Boundary to catch render-phase crashes that window.onerror misses.

Supabase Database Triggers

A Postgres trigger on api_call_logs could automatically call edge-remediation via pg_notify + a listener, or via net.http_post(), whenever a row with success = false is inserted. This removes the need for each edge function to explicitly fire-and-forget — the database itself becomes the error pipeline. Every failed API call, from any function, gets diagnosed automatically.

Slack Interactive Remediation

Instead of just alerting on Slack, the messages could include action buttons: "Retry Now", "Resolve", "View Details". Clicking "Retry Now" would call edge-remediation { action: "resolve-and-retry", id: "..." } directly from Slack — fixing issues from your phone without opening the Operations Center.

AI-Powered Root Cause Analysis

When errors repeat despite retries, feed the last N error entries for a company into Gemini/Claude: "These 5 sync failures over 2 hours all come from Company B. The errors went from rate_limit to timeout to auth_failure. What's the likely root cause?" The AI can correlate patterns humans might miss and suggest non-obvious fixes.

Health Score per Company

Aggregate remediation data into a per-company health score: error rate, avg resolution time, open issues. Surface this on the company dashboard and in the media company workspace. Flag companies trending toward degraded service before users notice.

Email/SMS Escalation Tiers

Slack is great for dev-team awareness, but for critical production issues (auth failures affecting all companies), the system could escalate beyond Slack: email the team lead after 30 minutes unresolved, SMS after 1 hour. Tier-based escalation ensures nothing falls through the cracks during off-hours.

The Big Picture

Today, when something breaks, the flow is: user notices → complains → developer investigates → fixes manually. That's reactive.

With this system, the flow becomes: error happens → diagnosed in milliseconds → auto-retried or Slack alert with exact fix steps → resolved before any user notices. That's proactive.
